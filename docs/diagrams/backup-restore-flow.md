# Логика резервного копирования и восстановления

Документ подробно описывает рабочую логику создания резервной копии и восстановления WordPress-сайта через Telegram-бота.

Связанные документы:

- `docs/specs/technical-spec.md` - основное техническое задание.
- `docs/specs/technical-spec-questions.md` - вопросы для уточнения требований.
- `docs/diagrams/backup-restore-flow.excalidraw` - визуальная схема процесса backup/restore.

## 1. Общие принципы

Бот управляет одним WordPress-сайтом на одном сервере. Все тяжелые операции выполняются локально на сервере через системные команды.

Тяжелыми операциями считаются:

- создание полного бэкапа;
- восстановление из бэкапа.

В один момент времени может выполняться только одна тяжелая операция.

```text
+-------------------+
| Telegram-команда  |
+---------+---------+
          |
          v
+-------------------+
| Проверка доступа  |
+---------+---------+
          |
          v
+-------------------+        занято        +----------------------+
| Проверка lock-а   |--------------------->| Отказать пользователю |
+---------+---------+                      | и показать операцию   |
          | свободно                       +----------------------+
          v
+-------------------+
| Запуск операции   |
+-------------------+
```

## 2. Общая модель операций

Каждая длительная операция записывается в таблицу `operations`.

Рекомендуемые статусы операции:

- `queued` - операция создана, но еще не выполняется;
- `running` - операция выполняется;
- `success` - операция завершена успешно;
- `failed` - операция завершена ошибкой;
- `cancelled` - операция отменена;
- `stale` - операция была `running`, но бот перезапустился и не может подтвердить ее состояние.

```text
queued
  |
  v
running
  |
  +--> success
  |
  +--> failed
  |
  +--> cancelled
  |
  +--> stale
```

При старте бот должен проверить операции в статусе `running`. Если активный системный процесс не найден или состояние нельзя надежно восстановить, такие операции следует пометить как `stale` или `failed` с причиной `bot restarted during operation`.

## 3. Блокировка тяжелых операций

Перед запуском backup/restore бот должен получить общий lock.

Рекомендуемая логика:

1. Проверить, нет ли активной операции `backup` или `restore` в локальной БД.
2. Получить file lock на уровне файловой системы.
3. Создать запись операции со статусом `running`.
4. Выполнить операцию.
5. Освободить lock в блоке `finally`.
6. Обновить статус операции на финальный.

```text
+--------------------------+
| Запрос backup/restore    |
+------------+-------------+
             |
             v
+--------------------------+
| Есть активная операция?  |
+------+-------------------+
       |
       +-- да --> отказ пользователю
       |
       +-- нет
             |
             v
+--------------------------+
| Получить file lock       |
+------------+-------------+
             |
             v
+--------------------------+
| Создать operation        |
| status = running         |
+------------+-------------+
             |
             v
+--------------------------+
| Выполнить операцию       |
+------------+-------------+
             |
             v
+--------------------------+
| success / failed         |
| освободить lock          |
+--------------------------+
```

## 4. Логика создания полного бэкапа

Полный бэкап включает:

- каталог `wordpress/` с файлами сайта;
- каталог `database/` с дампом БД `db.sql`;
- файл `metadata.json`;
- итоговый архив `{BACKUP__SITE_NAME}_backup_{YYYY-MM-DD_HH-mm-ss}.tar.zst`.

### 4.1. Основной поток backup

```text
+-----------------------+
| /backup или scheduler |
+-----------+-----------+
            |
            v
+-----------------------+
| Авторизация           |
+-----------+-----------+
            |
            v
+-----------------------+
| Проверка lock-а       |
+-----------+-----------+
            |
            v
+-----------------------+
| Preflight-проверки    |
+-----------+-----------+
            |
            v
+-----------------------+
| Создать staging       |
+-----------+-----------+
            |
            v
+-----------------------+
| Скопировать WORDPRESS__PATH   |
| в staging/wordpress   |
+-----------+-----------+
            |
            v
+-----------------------+
| Сделать mysqldump     |
| в database/db.sql     |
+-----------+-----------+
            |
            v
+-----------------------+
| Создать metadata.json |
+-----------+-----------+
            |
            v
+-----------------------+
| Упаковать .tar.zst    |
+-----------+-----------+
            |
            v
+-----------------------+
| Проверить архив       |
+-----------+-----------+
            |
            v
+-----------------------+
| Записать backups row  |
| status = available    |
+-----------+-----------+
            |
            v
+-----------------------+
| Уведомить Telegram    |
+-----------------------+
```

### 4.2. Preflight-проверки backup

Перед созданием бэкапа бот должен проверить:

1. Пользователь имеет роль `admin` или `superadmin`.
2. Нет активного backup/restore.
3. `WORDPRESS__PATH` существует и доступен для чтения.
4. `BACKUP__PATH_DIR` существует и доступен для записи.
5. Системные утилиты доступны: `tar`, `zstd`, `mysqldump`, `du`, `df`.
6. Настройки БД заполнены.
7. Свободного места больше `BACKUP__MIN_FREE_SPACE_GB`.
8. Staging-директория может быть создана внутри `BACKUP__TMP_PATH`.

Если любая проверка не проходит, операция не должна переходить к изменению файлов или созданию неполного архива.

### 4.3. Staging-структура

До упаковки архив собирается во временной staging-директории.

```text
staging/
├── wordpress/
│   ├── wp-content/
│   ├── wp-config.php
│   └── ...
├── database/
│   └── db.sql
└── metadata.json
```

Staging-директория должна удаляться после успешного завершения. После ошибки она может быть удалена автоматически или сохранена для диагностики, если это явно разрешено настройкой.

### 4.4. Метаданные бэкапа

Рекомендуемый состав `metadata.json`:

```json
{
  "schema_version": 1,
  "site_name": "example",
  "created_at": "2026-05-19T03:00:00+05:00",
  "backup_type": "full",
  "wordpress_path": "/var/www/site",
  "database": {
    "host": "localhost",
    "port": 3306,
    "name": "wordpress"
  },
  "content": {
    "wordpress_dir": "wordpress/",
    "database_dump": "database/db.sql"
  },
  "tool_versions": {
    "bot": "unknown",
    "python": "3.13",
    "mysql": "unknown",
    "tar": "unknown",
    "zstd": "unknown"
  }
}
```

Пароли, Telegram token и содержимое `.env` бота в `metadata.json` записывать нельзя.

### 4.5. Обработка ошибок backup

```text
+---------------------+
| Ошибка на этапе     |
+----------+----------+
           |
           v
+---------------------+
| Остановить процесс  |
+----------+----------+
           |
           v
+---------------------+
| Записать failed     |
| в operations        |
+----------+----------+
           |
           v
+---------------------+
| Записать failed     |
| в backups, если row |
| уже был создан      |
+----------+----------+
           |
           v
+---------------------+
| Очистить staging    |
+----------+----------+
           |
           v
+---------------------+
| Уведомить Telegram  |
| без секретов        |
+---------------------+
```

Сообщение пользователю должно быть коротким и безопасным. Подробности ошибки должны оставаться в локальном логе.

## 5. Логика восстановления

Восстановление является опасной операцией и доступно только роли `superadmin`.

Восстановление всегда требует подтверждения.

Источник восстановления:

- бэкап из локальной БД;
- внешний `.tar.zst` архив по локальному пути на сервере.

### 5.1. Основной поток restore

```text
+------------------------+
| /restore или           |
| /restore_path <path>   |
+-----------+------------+
            |
            v
+------------------------+
| Авторизация superadmin |
+-----------+------------+
            |
            v
+------------------------+
| Выбор архива           |
+-----------+------------+
            |
            v
+------------------------+
| Preflight-проверки     |
+-----------+------------+
            |
            v
+------------------------+
| Запрос подтверждения   |
+-----------+------------+
            |
            v
+------------------------+
| Проверка lock-а        |
+-----------+------------+
            |
            v
+------------------------+
| Аварийный бэкап        |
| текущего состояния     |
+-----------+------------+
            |
            v
+------------------------+
| Распаковка во temp     |
+-----------+------------+
            |
            v
+------------------------+
| Валидация структуры    |
+-----------+------------+
            |
            v
+------------------------+
| Восстановить файлы     |
+-----------+------------+
            |
            v
+------------------------+
| Восстановить БД        |
+-----------+------------+
            |
            v
+------------------------+
| Финальные проверки     |
+-----------+------------+
            |
            v
+------------------------+
| Записать результат     |
+-----------+------------+
            |
            v
+------------------------+
| Уведомить Telegram     |
+------------------------+
```

### 5.2. Preflight-проверки restore

Перед подтверждением восстановления бот должен проверить:

1. Пользователь имеет роль `superadmin`.
2. Выбранный бэкап существует.
3. Файл имеет ожидаемый формат `.tar.zst`.
4. Файл доступен для чтения.
5. Нет активного backup/restore.
6. `WORDPRESS__PATH` существует.
7. `BACKUP__PATH_DIR` доступен для записи аварийного бэкапа.
8. Доступны `tar`, `zstd`, `mysql`.
9. Настройки БД заполнены.
10. Достаточно свободного места для аварийного бэкапа и распаковки.

### 5.3. Валидация архива

Перед изменением текущего сайта бот должен распаковать архив во временную директорию и проверить структуру.

```text
restore-temp/
├── wordpress/
├── database/
│   └── db.sql
└── metadata.json
```

Обязательные проверки:

1. Есть директория `wordpress/`.
2. Есть файл `database/db.sql`.
3. Если есть `metadata.json`, он валиден как JSON.
4. В архиве нет абсолютных путей.
5. В архиве нет путей с `../`.
6. В архиве нет элементов, которые выходят за пределы временной директории при распаковке.
7. Размер распакованных данных не выглядит неожиданно опасным для текущего диска.

Если проверка не пройдена, восстановление должно остановиться до изменения текущего сайта.

### 5.4. Аварийный бэкап

Перед изменением сайта бот должен создать аварийный бэкап текущего состояния.

```text
BACKUP__PATH_DIR/
└── emergency_before_restore_{YYYY-MM-DD_HH-mm-ss}.tar.zst
```

Если аварийный бэкап не создан, восстановление должно быть остановлено.

Аварийный бэкап должен быть записан в `operations.details_json`, чтобы администратор мог найти его после ошибки.

### 5.5. Замена файлов

Рекомендуемый безопасный порядок:

```text
WORDPRESS__PATH                 -> текущий сайт
WORDPRESS__PATH.restore-old     -> временно сохраненный текущий сайт
WORDPRESS__PATH.restore-new     -> восстановленная версия из архива
```

Поток:

1. Подготовить `WORDPRESS__PATH.restore-new` из `restore-temp/wordpress`.
2. Проверить, что новая директория содержит ожидаемые файлы WordPress.
3. Переименовать текущий `WORDPRESS__PATH` в `WORDPRESS__PATH.restore-old`.
4. Переименовать `WORDPRESS__PATH.restore-new` в `WORDPRESS__PATH`.
5. Если шаг 4 не удался, попытаться вернуть `WORDPRESS__PATH.restore-old` обратно в `WORDPRESS__PATH`.

```text
+-----------------------------+
| Подготовить restore-new     |
+-------------+---------------+
              |
              v
+-----------------------------+
| Проверить restore-new       |
+-------------+---------------+
              |
              v
+-----------------------------+
| WORDPRESS__PATH -> restore-old      |
+-------------+---------------+
              |
              v
+-----------------------------+
| restore-new -> WORDPRESS__PATH      |
+-------------+---------------+
              |
              v
+-----------------------------+
| Проверить WORDPRESS__PATH           |
+-----------------------------+
```

Удалять старую директорию сразу нельзя. Ее нужно сохранять до успешного завершения всей операции или до ручной очистки.

### 5.6. Восстановление БД

После замены файлов бот импортирует `database/db.sql` через `mysql`.

Логика:

1. Проверить наличие `database/db.sql`.
2. Подготовить безопасный способ передачи пароля.
3. Выполнить импорт через `mysql` без shell.
4. Проверить код возврата.
5. Записать результат в локальную БД.

Если импорт БД завершился ошибкой после замены файлов, операция должна быть помечена как `failed`, а пользователь должен получить путь к аварийному бэкапу. Автоматический rollback должен быть либо явно реализован, либо явно запрещен в ТЗ.

## 6. Ошибки восстановления и rollback

Восстановление должно быть спроектировано как последовательность точек невозврата.

```text
До аварийного бэкапа:
  можно безопасно остановиться

После аварийного бэкапа, до замены файлов:
  можно безопасно остановиться

После замены файлов, до импорта БД:
  сайт уже изменен частично

После импорта БД:
  сайт изменен полностью или частично
```

Рекомендуемая таблица поведения:

| Этап ошибки | Сайт изменен | Действие |
|---|---:|---|
| Preflight | Нет | Остановить, сообщить причину |
| Подтверждение | Нет | Остановить без ошибки |
| Аварийный бэкап | Нет | Остановить, восстановление не начинать |
| Распаковка архива | Нет | Остановить, удалить temp |
| Валидация архива | Нет | Остановить, удалить temp |
| Подготовка restore-new | Нет | Остановить, удалить temp |
| Переименование текущего WORDPRESS__PATH | Частично возможно | Попытаться вернуть старый каталог |
| Замена файлов завершена, импорт БД упал | Да | Сообщить failed, указать аварийный бэкап |
| Импорт БД завершен, финальная проверка упала | Да | Сообщить warning/failed, указать аварийный бэкап |

## 7. Telegram-уведомления

### 7.1. Backup

Успешный бэкап:

```text
Бэкап создан успешно.
Файл: /path/to/site_backup_2026-05-19_03-00-00.tar.zst
Размер: 2.4 GB
```

Ошибка бэкапа:

```text
Не удалось создать бэкап.
Причина: недостаточно свободного места.
Подробности см. в локальных логах.
```

### 7.2. Restore

Перед подтверждением:

```text
Вы собираетесь восстановить сайт из бэкапа:
/path/to/backup.tar.zst

Перед восстановлением будет создан аварийный бэкап текущего состояния.
Подтвердить восстановление?
```

Успешное восстановление:

```text
Сайт восстановлен успешно.
Источник: /path/to/backup.tar.zst
Аварийный бэкап: /path/to/emergency_before_restore_2026-05-19_03-00-00.tar.zst
```

Ошибка восстановления:

```text
Восстановление завершилось ошибкой.
Этап: импорт базы данных.
Аварийный бэкап: /path/to/emergency_before_restore_2026-05-19_03-00-00.tar.zst
Подробности см. в локальных логах.
```

## 8. Минимальные инварианты

Эти условия должны оставаться истинными независимо от успешности операции:

1. Бот не запускает два backup/restore одновременно.
2. Пароли и Telegram token не попадают в Telegram-сообщения.
3. Пароли и Telegram token не попадают в логи.
4. Восстановление не меняет текущий сайт до успешной валидации архива.
5. Восстановление не начинается, если аварийный бэкап не создан.
6. Все временные директории либо очищены, либо явно указаны в логах.
7. Любая операция имеет финальный статус в локальной БД.
8. Пользователь получает понятный результат операции.
