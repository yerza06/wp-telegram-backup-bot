# Стратегия тестирования Telegram-бота для WordPress Backup

## 1. Контекст

Проект представляет собой Telegram-бота для управления резервным копированием и восстановлением одного WordPress-сайта.

Согласно техническому заданию, бот устанавливается на тот же сервер, где работает WordPress, и выполняет операции локально через системные утилиты:

- `tar`
- `zstd`
- `mysqldump`
- `mysql`
- `df`
- `du`
- `wp`
- `runuser`

Главная и самая рискованная функция проекта — создание полного бэкапа сайта и восстановление сайта из бэкапа. Поэтому тестирование нужно разделить на безопасные локальные проверки и интеграционные проверки на отдельном тестовом сервере.

## 2. Принятое решение

Тестирование будет разделено на два уровня:

```text
Локальная машина разработчика
└── тестирование логики бота, Telegram-команд и прав доступа

Тестовый VPS-сервер
└── тестирование реального WordPress, backup и restore
```

Такой подход позволяет безопасно разрабатывать бота локально, но при этом проверять критичные операции в окружении, максимально похожем на production.

## 3. Что тестируем локально

Локально тестируются части проекта, которые не требуют настоящего WordPress-сайта, реальной MariaDB и операций с системными файлами сайта.

### 3.1. Авторизация и роли пользователей

Локально проверяются:

- доступ неизвестного пользователя;
- доступ `viewer`;
- доступ `admin`;
- доступ `superadmin`;
- запрет опасных команд для неподходящих ролей;
- разрешение команд для правильных ролей.

Примеры проверок:

- неизвестный пользователь не может пользоваться ботом;
- `viewer` может смотреть статус и список бэкапов;
- `viewer` не может запускать `/backup` и `/restore`;
- `admin` может запускать `/backup`;
- только разрешенные роли могут запускать опасные операции.

### 3.2. Telegram-интерфейс

Локально проверяются:

- команды `/start`;
- `/help`;
- `/status`;
- `/backup` без реального запуска backup-сервиса;
- `/restore` без реального восстановления;
- inline-кнопки;
- сообщения об ошибках;
- подтверждения опасных действий.

### 3.3. Логика приложения

Локально проверяются:

- обработка конфигурации из `.env`;
- работа с локальной SQLite-БД;
- запись операций в таблицу `operations`;
- запись метаданных бэкапов в таблицу `backups`;
- блокировка параллельных операций;
- обработка неуспешных операций;
- форматирование ответов для Telegram.

### 3.4. Mock/Fake для системных команд

Для локальных тестов реальные команды `mysqldump`, `mysql`, `tar`, `zstd`, `wp` не должны выполнять опасные действия.

Рекомендуется использовать слой запуска команд, например `CommandRunner`, который в тестах можно заменить на fake/mock-реализацию.

Это позволит проверять, что бот правильно строит команды и обрабатывает результат, не затрагивая настоящий WordPress.

## 4. Что тестируем на тестовом VPS

На тестовом VPS поднимается отдельный WordPress-сайт, максимально похожий на production-сайт.

На этом сервере проверяются операции, которые зависят от реального окружения Ubuntu, NGINX, PHP, MariaDB, прав файлов и системных утилит.

### 4.1. Окружение тестового сервера

На тестовом VPS должны быть установлены:

- Ubuntu Server;
- NGINX;
- PHP нужной версии;
- MariaDB или MySQL;
- WordPress;
- WP-CLI;
- `tar`;
- `zstd`;
- `mysqldump`;
- `mysql`;
- `runuser`;
- systemd service для запуска бота.

Желательно, чтобы структура директорий была похожа на production:

```text
/var/www/example-site
/var/backups/example-site
/tmp/example-site-backup
```

### 4.2. Backup

На тестовом сервере проверяются:

- запуск команды `/backup`;
- создание дампа базы данных через `mysqldump`;
- архивирование файлов WordPress;
- сжатие архива через `zstd`;
- сохранение архива в директорию бэкапов;
- запись информации о бэкапе в SQLite;
- корректный статус операции;
- поведение при нехватке свободного места;
- поведение при ошибке `mysqldump`;
- поведение при ошибке `tar` или `zstd`.

### 4.3. Restore

На тестовом сервере проверяются:

- список доступных бэкапов;
- выбор бэкапа для восстановления;
- обязательное подтверждение восстановления;
- восстановление файлов WordPress;
- восстановление базы данных через `mysql`;
- корректность прав файлов после восстановления;
- работоспособность сайта после восстановления;
- обработка ошибки восстановления;
- запрет параллельного запуска backup/restore.

Restore не должен тестироваться сразу на production-сайте.

### 4.4. WP-CLI и обслуживание

На тестовом сервере проверяются:

- команда `/clear_cache`;
- запуск WP-CLI от нужного пользователя;
- работа `runuser`;
- корректность прав пользователя `www-data`;
- сообщения об ошибках при недоступности WP-CLI.

### 4.5. Диск и мониторинг

На тестовом сервере проверяются:

- команда `/disk`;
- команда `/disk_chart`;
- получение данных через `df`;
- получение размера директорий через `du`;
- реакция на малый объем свободного места.

## 5. Отдельный Telegram-бот для staging

Для тестового VPS рекомендуется создать отдельного Telegram-бота через BotFather.

```text
Production bot: @project_backup_bot
Staging bot:    @project_backup_test_bot
```

Это снижает риск случайного запуска опасной команды на production-сервере.

Также рекомендуется явно отображать окружение в сообщениях бота:

```text
🧪 STAGING BOT
Сайт: staging.example.com
```

Для production:

```text
🚨 PRODUCTION BOT
Сайт: example.com
```

## 6. Production-сервер

На production-сервере не нужно тестировать опасные сценарии до прохождения проверок на staging.

На production допустимо проверять:

- `/start`;
- `/help`;
- `/status`;
- `/disk`;
- `/backups`;
- создание backup после успешных тестов на staging.

Восстановление на production должно выполняться только после полной проверки restore на тестовом VPS.

Перед restore на production желательно автоматически создавать emergency backup текущего состояния сайта.

## 7. Итоговая схема

```text
Локально:
- роли и права доступа;
- Telegram-команды;
- inline-кнопки;
- SQLite;
- бизнес-логика;
- обработка ошибок;
- fake/mock системных команд.

Тестовый VPS:
- настоящий WordPress;
- настоящий NGINX/PHP/MariaDB;
- mysqldump/mysql;
- tar/zstd;
- WP-CLI;
- systemd;
- backup;
- restore;
- права файлов;
- проверка сайта после восстановления.

Production:
- только проверенные сценарии;
- restore только после успешного staging-теста.
```

## 8. Вывод

Выбранная стратегия является безопасной и практичной.

Локальная среда используется для быстрой разработки и проверки логики бота. Тестовый VPS используется для проверки реальных операций с WordPress, базой данных и файловой системой. Production-сервер используется только после успешной проверки на staging.
