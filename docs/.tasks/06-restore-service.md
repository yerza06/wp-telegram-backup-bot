# Задача 06. Сервис восстановления

## Цель

Реализовать восстановление сайта из бэкапа с обязательным подтверждением и аварийным бэкапом текущего состояния.

## Объем работ

- Реализовать `/restore` со списком доступных бэкапов из локальной БД.
- Реализовать `/restore_<id>` для выбора бэкапа.
- Реализовать `/restore_path <path>` для внешнего локального архива.
- Добавить обязательное подтверждение через inline-кнопку или confirmation command.
- Перед восстановлением:
  - проверить отсутствие активного `backup`/`restore`;
  - проверить существование архива;
  - проверить расширение и формат `.tar.zst`;
  - создать аварийный бэкап текущего состояния;
  - проверить свободное место;
  - распаковать архив во временную директорию внутри `BACKUP__TMP_PATH`;
  - проверить наличие `wordpress/` и `database/db.sql`.
- Восстановить файлы WordPress через временную директорию.
- Восстановить БД через `mysql`.
- Записать результат в `operations`.
- Обновить статус бэкапа на `restored`, если восстановление выполнялось из локальной БД.

## Ожидаемые файлы

- `bot/services/restore.py`
- `bot/handlers/restore.py`
- `bot/keyboards/restore.py`

## Критерии готовности

- Восстановление не запускается без подтверждения.
- При невалидном внешнем архиве текущий сайт не изменяется.
- Перед восстановлением создается аварийный бэкап.
- Веб-сервер и PHP-FPM не останавливаются.
- Восстановление не стартует параллельно с backup/restore.

## Проверки

- Тест валидации структуры архива.
- Тест отказа без подтверждения.
- Тест отказа при активной тяжелой операции.
- Интеграционная проверка восстановления во временные директории.
