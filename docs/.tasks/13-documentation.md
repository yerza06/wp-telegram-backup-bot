# Задача 13. Документация проекта

## Цель

Оформить документацию проекта в `docs/` так, чтобы рабочие инструкции, ТЗ, вопросы и диаграммы были разнесены по понятным директориям.

## Объем работ

- Поддерживать структуру:
  - `docs/README.md` - индекс документации;
  - `docs/specs/` - ТЗ и вопросы к требованиям;
  - `docs/diagrams/` - схемы и исходники диаграмм;
  - `docs/.tasks/` - план реализации и task-файлы;
  - `docs/acceptance-checklist.md` - приемочный чеклист.
- Обновить ссылки после перемещения документов.
- Добавить краткую инструкцию по локальному запуску.
- Добавить описание основных Telegram-команд.
- Добавить описание формата бэкапа:
  - `wordpress/`;
  - `database/db.sql`;
  - `metadata.json`.
- Добавить раздел диагностики:
  - проверка `.env`;
  - проверка доступа к WordPress;
  - проверка `mysqldump`/`mysql`;
  - проверка `wp` и `runuser`;
  - проверка свободного места.

## Ожидаемые файлы

- `docs/README.md`
- `docs/acceptance-checklist.md`
- `README.md`
- `docs/specs/`
- `docs/diagrams/`
- `docs/.tasks/`

## Критерии готовности

- В `docs/README.md` есть актуальный индекс документов.
- ТЗ и вопросы к ТЗ лежат в `docs/specs/`.
- Диаграммы лежат в `docs/diagrams/`.
- Task-файлы лежат в `docs/.tasks/`.
- В документации нет устаревших ссылок на старые пути.

## Проверки

- Проверить, что в документации нет ссылок на старое расположение ТЗ, вопросов и диаграмм в корне `docs/`.
- Ручная проверка ссылок из `docs/README.md`.
