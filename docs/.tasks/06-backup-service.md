# Задача 06. Сервис полного резервного копирования

## Цель

Реализовать создание полного бэкапа WordPress в формате одного `.tar.zst` архива.

## Объем работ

- Перед запуском проверить:
  - наличие `WORDPRESS__PATH`;
  - доступность `BACKUP__PATH_DIR`;
  - доступность `BACKUP__TMP_PATH`;
  - свободное место не меньше `BACKUP__MIN_FREE_SPACE_GB`;
  - отсутствие активного `backup` или `restore`.
- Создать запись `operations` со статусом `running`.
- Создать запись `backups`.
- Создать временную staging-директорию внутри `BACKUP__TMP_PATH`.
- Скопировать весь `WORDPRESS__PATH` в staging как обычную папку `wordpress/`.
- Выполнить `mysqldump` и сохранить результат в `database/db.sql`.
- Создать `metadata.json`.
- Собрать итоговый архив:
  - имя `{BACKUP__SITE_NAME}_backup_{YYYY-MM-DD_HH-mm-ss}.tar.zst`;
  - структура внутри: `wordpress/`, `database/db.sql`, `metadata.json`.
- Использовать `tar --zstd` или безопасный эквивалент без shell.
- Обновить записи `backups` и `operations`.
- Отправить итоговое Telegram-сообщение с локальным путем к архиву.
- При ошибке отправлять безопасное сообщение без секретов и длинного stderr.

## Ожидаемые файлы

- `bot/services/backup.py`
- `bot/services/process.py`
- `bot/services/filesystem.py`
- `bot/services/disk.py`
- `bot/utils/sanitize.py`

## Критерии готовности

- Через Telegram доступна только команда полного бэкапа.
- Нет команд для отдельного файлового или DB-only бэкапа.
- SQL-дамп не создается как отдельный `.sql.gz`.
- Готовый бэкап не отправляется файлом в Telegram.
- Пароль БД не передается в аргументах команд, Telegram-сообщениях или логах.

## Проверки

- Интеграционный тест на временном каталоге WordPress.
- Проверка структуры архива через `tar -tf`.
- Проверка, что в операции сохраняется `success` или `failed`.
- Проверка отказа при активной тяжелой операции.
