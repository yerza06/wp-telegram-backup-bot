# Задача 08. Восстановление из внешнего архива

## Цель

Реализовать восстановление сайта из внешнего локального архива по пути на сервере с обязательной проверкой целостности до изменения текущего сайта.

## Объем работ

- Реализовать `/restore_path <path>` для внешнего локального архива.
- Добавить обязательное подтверждение через inline-кнопку или confirmation command.
- Перед восстановлением:
  - проверить отсутствие активного `backup`/`restore`;
  - проверить существование архива;
  - проверить расширение `.tar.zst`;
  - проверить, что архив читается и распаковывается;
  - проверить наличие `wordpress/`;
  - проверить наличие `database/db.sql`;
  - проверить наличие и корректность `metadata.json`, если он есть;
  - проверить свободное место для распаковки и аварийного бэкапа;
  - создать аварийный бэкап текущего состояния.
- Распаковывать архив только во временную директорию внутри `BACKUP__TMP_PATH`.
- Не менять текущий WordPress, если проверка архива не пройдена.
- Восстановить файлы WordPress через временную директорию.
- Восстановить БД через `mysql`.
- Записать результат в `operations`.
- Не создавать запись в `backups` для внешнего архива, если он не является результатом работы текущего бота.

## Ожидаемые файлы

- `bot/services/restore.py`
- `bot/services/archive.py`
- `bot/handlers/restore.py`
- `bot/keyboards/restore.py`

## Критерии готовности

- При невалидном внешнем архиве текущий сайт не изменяется.
- Проверка целостности выполняется до аварийного изменения файлов или БД.
- Восстановление не запускается без подтверждения.
- Восстановление не стартует параллельно с backup/restore.
- Пользователь получает безопасное сообщение без длинного stderr и секретов.

## Проверки

- Тест отказа для несуществующего архива.
- Тест отказа для архива без `wordpress/`.
- Тест отказа для архива без `database/db.sql`.
- Тест отказа для поврежденного `.tar.zst`.
- Интеграционная проверка восстановления во временные директории.
