# Техническое задание: Telegram-бот для резервного копирования WordPress

## 1. Назначение

Разработать локальный Telegram-бот для управления резервным копированием одного WordPress-сайта и мониторинга дискового пространства сервера.

Бот устанавливается на тот же сервер, где работает WordPress, и выполняет операции локально через дочерние процессы:

- `tar`
- `zstd`
- `mysqldump`
- `mysql`
- `df -h`
- `du`
- `cp`
- `wp`
- `runuser`

Один экземпляр бота обслуживает только один сайт на одном сервере. Для нескольких сайтов запускаются несколько независимых экземпляров бота, каждый со своим `.env`, своей локальной БД и своей конфигурацией.

## 2. Цели

- Создание полного бэкапа сайта: файлы + база данных.
- Восстановление сайта из резервной копии.
- Просмотр списка доступных резервных копий.
- Очистка кэша WordPress через WP-CLI.
- Мониторинг свободного места на диске.
- Просмотр графика свободного и занятого места на диске через Telegram.
- Управление операциями через Telegram.
- Ограничение доступа к боту только разрешенным пользователям.

## 3. Не цели

- Управление несколькими сайтами из одного экземпляра бота.
- Удаленное подключение к другим серверам по SSH.
- Облачное хранение резервных копий на первом этапе.
- Прямая интеграция с Google Drive на текущем этапе.
- Web-интерфейс.
- Автоматическое определение всех параметров WordPress без конфигурации.
- Управление пользователями и ролями через Telegram-команды.
- Управление расписанием бэкапов через Telegram-команды.
- Отдельная команда проверки целостности бэкапа в текущей версии.
- Поддержка WordPress multisite без отдельного уточнения.

## 4. Платформа и окружение

Платформа:

- Ubuntu Server.
- Python 3.13 или версия, указанная в `pyproject.toml`.
- WordPress, установленный напрямую на том же сервере без Docker, виртуальной машины или контейнера.
- MySQL или MariaDB, работающие напрямую на том же сервере без Docker.
- Установленные системные утилиты: `tar`, `zstd`, `mysqldump`, `mysql`, `df`, `du`, `cp`, `wp`, `runuser`.

Бот должен запускаться как долгоживущий процесс. Рекомендуемый способ запуска в production: `systemd` service.

## 5. Конфигурация

Конфигурация хранится в `.env` файле конкретного экземпляра бота.

Минимальный набор переменных:

```env
TELEGRAM__BOT_TOKEN="XXXXXXXXX:XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX"
TELEGRAM__SUPERADMIN_USERS=[]
TELEGRAM__ADMIN_USERS=[]
TELEGRAM__VIEWER_USERS=[]

WORDPRESS__PATH=
BACKUP__PATH_DIR=
BACKUP__TMP_PATH=

WORDPRESS__DB_HOST=localhost
WORDPRESS__DB_PORT=3306
WORDPRESS__DB_NAME=
WORDPRESS__DB_USER=
WORDPRESS__DB_PASSWORD=

BACKUP__SITE_NAME=
BACKUP__TIMEZONE=Asia/Almaty

BACKUP__MIN_FREE_SPACE_GB=

CORE__SQLITE_PATH=bot.sqlite3
CORE__LOGS_DIR=logs
```

Дополнительные переменные, которые могут понадобиться:

```env
MYSQLDUMP_PATH=mysqldump
MYSQL_PATH=mysql
TAR_PATH=tar
ZSTD_PATH=zstd
DF_PATH=df
DU_PATH=du
CP_PATH=cp
WP_CLI_PATH=wp
RUNUSER_PATH=runuser
WORDPRESS__CLI_RUN_AS_USER=www-data

BACKUP__FILE_PREFIX=

DISK_CHART__ENABLED=true
DISK_CHART__FORMAT=png

SCHEDULE__ENABLED=false
SCHEDULE__BACKUP_CRON="0 3 * * 5"
```

## 6. Локальная БД бота

Бот использует отдельную локальную БД для хранения метаданных и истории операций. Предпочтительный вариант для одного локального экземпляра: SQLite.

### 6.1. Таблица `backups`

Хранит информацию о созданных резервных копиях.

Поля:

- `id`
- `status`: `created`, `failed`, `restored`
- `created_at`
- `finished_at`
- `archive_path`
- `file_size_bytes`
- `is_removed`: `false` по умолчанию, `true` если архив удален с сервера через бота
- `error_message`

### 6.2. Таблица `operations`

Хранит историю команд и длительных операций.

Поля:

- `id`
- `operation_type`: `backup`, `restore`, `disk_check`, `cache_clear`
- `status`: `queued`, `running`, `success`, `failed`, `cancelled`
- `started_at`
- `finished_at`
- `telegram_user_id`
- `backup_id`
- `details_json`
- `error_message`

## 7. Telegram-команды

### 7.1. Базовые команды

- `/start` - показать краткое меню и статус доступа.
- `/help` - показать список команд.
- `/status` - показать состояние бота, текущую операцию, путь сайта, папку бэкапов, последний бэкап и свободное место.
- `/disk` - показать результат `df -h` по целевому разделу.
- `/disk_chart` - отправить график занятого и свободного места на диске.

### 7.2. Команды резервного копирования

- `/backup` - создать полный бэкап.
- `/backups` - показать список последних резервных копий.

Пользователь не должен выбирать между бэкапом только файлов и бэкапом только базы данных. Через Telegram доступно создание только полного бэкапа, который включает файлы WordPress и дамп базы данных.

### 7.3. Команды восстановления

- `/restore` - показать список доступных бэкапов для восстановления.
- `/restore_<id>` - восстановить сайт из выбранного бэкапа.
- `/restore_path <path>` - восстановить сайт из внешнего бэкапа по локальному пути на сервере.

Восстановление должно требовать подтверждения через inline-кнопку или отдельную команду подтверждения.

Если бот уже создает бэкап или выполняет восстановление, команды восстановления должны быть недоступны до завершения текущей операции.

### 7.4. Команды обслуживания

- `/clear_cache` - очистить кэш WordPress через WP-CLI.

## 8. Telegram-интерфейс

Бот должен поддерживать inline-кнопки для основных действий:

- Создать полный бэкап.
- Проверить диск.
- Показать график диска.
- Показать бэкапы.
- Очистить кэш WordPress.

Если бот находится в статусе создания бэкапа или восстановления, кнопки запуска бэкапа и восстановления должны либо скрываться, либо отвечать сообщением о текущей операции без запуска новой операции.

Для опасных операций обязательны подтверждения:

- Восстановление.
- Очистка кэша WordPress, если команда может повлиять на работу сайта.

## 9. Авторизация

Доступ разрешен только Telegram-пользователям, указанным в переменных окружения:

- `TELEGRAM__SUPERADMIN_USERS`
- `TELEGRAM__ADMIN_USERS`
- `TELEGRAM__VIEWER_USERS`

Идентификаторы пользователей хранятся только в `.env` или переменных окружения. Бот не должен поддерживать добавление, удаление или изменение пользователей через Telegram-команды.

Роли:

- `superadmin` - полный доступ ко всем командам, включая восстановление и очистку кэша.
- `admin` - доступ к созданию бэкапа, просмотру статуса, диска, графика, списка бэкапов и очистке кэша. Восстановление требует роли `superadmin`.
- `viewer` - доступ только к просмотру статуса, свободного места, графика диска и списка бэкапов.

Если один Telegram ID указан в нескольких списках ролей, применяется самая высокая роль: `superadmin` выше `admin`, `admin` выше `viewer`.

Если пользователь не разрешен:

- бот не выполняет команды;
- бот отвечает коротким сообщением об отказе;
- попытка доступа записывается в лог.

Если пользователь разрешен, но его роль не имеет права на команду:

- бот не выполняет команду;
- бот отвечает коротким сообщением о недостатке прав;
- попытка записывается в лог.

## 10. Создание резервной копии

### 10.1. Этап архивации файлов

Этот этап выполняется только внутри полного бэкапа и не доступен пользователю как отдельная команда.

Архивироваться должен весь каталог WordPress из `WORDPRESS__PATH`, а не только `wp-content`. В архив должны попадать конфигурационные файлы сайта, включая `wp-config.php` и `.env`, если они находятся внутри `WORDPRESS__PATH`.

Бот должен:

1. Проверить наличие `WORDPRESS__PATH`.
2. Проверить доступность `BACKUP__PATH_DIR`.
3. Проверить свободное место.
4. Перевести операцию в статус `running`, чтобы бот считался занятым созданием бэкапа.
5. Создать временную staging-директорию полного бэкапа внутри `BACKUP__TMP_PATH`.
6. Скопировать файлы WordPress в staging-директорию как обычную папку `wordpress/`.
7. Записать метаданные этапа в локальную БД.

Исключения из архива не поддерживаются в текущей версии. В бэкап должен попадать весь каталог WordPress из `WORDPRESS__PATH`.

Внутри итогового архива файлы WordPress не должны быть вложенным архивом. Они должны лежать как обычная директория:

```text
wordpress/
```

### 10.2. Этап дампа базы данных

Этот этап выполняется только внутри полного бэкапа и не доступен пользователю как отдельная команда.

Бот должен:

1. Проверить доступность настроек БД.
2. Перевести операцию в статус `running`, чтобы бот считался занятым созданием бэкапа.
3. Выполнить `mysqldump`.
4. Сохранить дамп в staging-директорию полного бэкапа как `database/db.sql`.
5. Записать метаданные этапа в локальную БД.

Формат дампа внутри итогового архива:

```text
database/db.sql
```

SQL-дамп не должен быть отдельным `.sql.gz` файлом. Он сжимается вместе со всем полным бэкапом внутри итогового `.tar.zst` архива. Пароль БД не должен попадать в аргументы команд, Telegram-сообщения или логи.

### 10.3. Полный бэкап

Полный бэкап должен включать:

- обычную папку `wordpress/` с файлами WordPress;
- обычную папку `database/` с дампом базы данных `db.sql`;
- файл метаданных.

Формат имени итогового архива:

```text
{BACKUP__SITE_NAME}_backup_{YYYY-MM-DD_HH-mm-ss}.tar.zst
```

Полный бэкап должен храниться как один итоговый `.tar.zst` архив.

Структура внутри архива:

```text
{BACKUP__SITE_NAME}_backup_{YYYY-MM-DD_HH-mm-ss}.tar.zst
├── wordpress/
│   ├── wp-content/
│   ├── wp-config.php
│   └── ...
├── database/
│   └── db.sql
└── metadata.json
```

Для сжатия итогового архива должен использоваться `zstd`, например через `tar --zstd`, если установленная версия `tar` это поддерживает. Если `tar --zstd` недоступен, реализация должна использовать эквивалентный безопасный способ создания `.tar.zst` без shell-инъекций.

При запуске полного бэкапа бот должен создать запись операции со статусом `running` и держать этот статус до полного завершения всех этапов: архива файлов, дампа БД и записи метаданных.

Готовый бэкап не отправляется файлом в Telegram. Результат в Telegram отправляется по итогам полного бэкапа, а не после каждого внутреннего этапа отдельно. В сообщении должен быть полный локальный путь к созданному бэкапу, чтобы администратор мог скачать его с сервера самостоятельно, например через `scp`.

После успешного или неуспешного бэкапа бот должен отправить уведомление в Telegram с результатом операции. При ошибке сообщение должно быть безопасным: без паролей, секретов и длинного stderr, но с понятной причиной и ссылкой на локальные логи.

Отдельные пользовательские команды для создания только файлового архива или только дампа базы данных не реализуются.

## 11. Восстановление

Восстановление является опасной операцией и требует явного подтверждения.

Восстановление не может запускаться, если уже выполняется создание бэкапа или другое восстановление.

Перед восстановлением бот должен:

1. Проверить выбранный бэкап из локальной БД или внешний бэкап по указанному локальному пути на сервере.
2. Создать аварийный бэкап текущего состояния.
3. Остановить выполнение, если свободного места недостаточно.
4. Распаковать `.tar.zst` во временную директорию внутри `BACKUP__TMP_PATH`.
5. Проверить наличие ожидаемой структуры `wordpress/` и `database/db.sql`.
6. Восстановить файлы WordPress из временной директории и затем заменить текущую директорию сайта.
7. Восстановить базу данных из `database/db.sql` через `mysql`.
8. Записать результат операции в локальную БД.
9. Отправить результат пользователю.

Восстановление внешнего бэкапа нужно для сценария, когда старый бэкап скачан из Google Drive или другого внешнего хранилища и затем скопирован на сервер через `scp` или `rsync`.

Для внешнего бэкапа пользователь должен указать локальный путь на сервере. Бот не должен загружать внешний бэкап через Telegram.

Внешний бэкап должен соответствовать тому же формату:

```text
*.tar.zst
├── wordpress/
└── database/
    └── db.sql
```

Если внешний архив не соответствует формату, восстановление должно быть остановлено до изменения текущего сайта.

Перед восстановлением не требуется останавливать веб-сервер или PHP-FPM.

## 12. Мониторинг диска

Команда `/disk` должна показывать:

- общий размер раздела;
- занято;
- свободно;
- процент использования;
- путь, для которого выполнена проверка.

Команда `/disk_chart` должна:

1. Получить информацию о диске через безопасный вызов `df`.
2. Получить размер каталога WordPress из `WORDPRESS__PATH`.
3. Получить размер каталога бэкапов из `BACKUP__PATH_DIR`.
4. Построить изображение графика через `matplotlib`.
5. Отправить PNG-изображение пользователю в Telegram.
6. Записать операцию проверки диска в локальную БД.

График должен быть круговой диаграммой в виде пирога. Диаграмма должна показывать:

- сколько места использует сервер без учета `WORDPRESS__PATH` и `BACKUP__PATH_DIR`;
- сколько места использует сам сайт WordPress;
- сколько места используют резервные копии;
- сколько места осталось свободным.

Перед созданием бэкапа бот должен проверять свободное место и предупреждать пользователя, если свободного места меньше `BACKUP__MIN_FREE_SPACE_GB`.

## 13. Очистка кэша WordPress

Бот должен поддерживать очистку кэша WordPress через WP-CLI.

Команда `/clear_cache` должна:

1. Проверить наличие `WORDPRESS__PATH`.
2. Проверить доступность `wp` или пути из `WP_CLI_PATH`.
3. Выполнить очистку кэша через WP-CLI в директории WordPress от имени пользователя `www-data` или пользователя из `WORDPRESS__CLI_RUN_AS_USER`.
4. Записать результат операции в локальную БД.
5. Отправить пользователю результат в Telegram.

Команды очистки:

```text
wp cache flush --path={WORDPRESS__PATH}
wp transient delete --all --path={WORDPRESS__PATH}
```

Команда должна выполняться без shell-инъекций, через список аргументов.

Так как бот может запускаться от имени `root`, WP-CLI команды должны выполняться от имени `www-data`, чтобы права файлов и поведение WordPress соответствовали веб-окружению сайта.

## 14. Планировщик

Бот должен поддерживать автоматическое создание полного бэкапа по расписанию через `APScheduler`.

Настройки расписания задаются только через конфигурацию и переменные окружения. Бот не должен поддерживать изменение расписания через Telegram-команды.

Конфигурация должна поддерживать cron-подобное управление расписанием:

- `SCHEDULE__ENABLED` - включает или выключает планировщик;
- `SCHEDULE__BACKUP_CRON` - cron-выражение для расписания;
- `BACKUP__TIMEZONE` - часовой пояс выполнения расписания.

Расписание по умолчанию: каждую пятницу в 03:00 по `BACKUP__TIMEZONE`.

```env
SCHEDULE__BACKUP_CRON="0 3 * * 5"
```

Плановый бэкап должен использовать тот же сервис полного бэкапа, что и команда `/backup`, и соблюдать общий запрет на параллельные тяжелые операции.

Если в момент планового запуска уже идет бэкап или восстановление, плановая операция должна быть отложена. Бот должен отправить предупреждение администраторам в Telegram, записать событие в локальную БД и логи, а затем выполнить плановый бэкап после завершения текущей тяжелой операции.

После успешного или неуспешного планового бэкапа бот должен отправить уведомление в Telegram разрешенным администраторам.

## 15. Логирование

Бот должен вести логи:

- запуск и остановка;
- входящие команды;
- отказ в доступе;
- запуск операции;
- завершение операции;
- запуск и отложение плановых операций;
- ошибки дочерних процессов;
- stderr системных команд;
- результаты очистки кэша WordPress.

Логи должны храниться локально, например:

```text
logs/bot.log
```

Подробный журнал операций хранится только локально и не показывается через Telegram.

## 16. Обработка ошибок

Бот должен корректно обрабатывать:

- отсутствие системной утилиты;
- неверный путь WordPress;
- недоступную папку бэкапов;
- ошибку `mysqldump`;
- ошибку `mysql`;
- нехватку свободного места;
- отсутствие прав на чтение или запись;
- поврежденный или неполный бэкап;
- параллельный запуск двух тяжелых операций;
- отсутствие WP-CLI;
- ошибку запуска команды от имени `www-data`;
- ошибку выполнения `wp cache flush`;
- ошибку выполнения `wp transient delete --all`;
- ошибку создания итогового `.tar.zst` архива;
- неверный формат внешнего `.tar.zst` бэкапа;
- ошибку определения размера `WORDPRESS__PATH` или `BACKUP__PATH_DIR`;
- ошибку построения графика диска через `matplotlib`.

Одновременно должна выполняться только одна операция резервного копирования или восстановления.

Если бот уже находится в статусе создания бэкапа или восстановления, пользователь не может запустить:

- новый полный бэкап;
- восстановление из бэкапа.

В этом случае бот должен отказать в запуске новой тяжелой операции, показать текущий статус операции и коротко сообщить, что нужно дождаться завершения. Это требуется, чтобы параллельные операции не повредили файлы WordPress или базу данных.

## 17. Безопасность

- Пароли БД не должны попадать в Telegram-сообщения.
- Пароли БД не должны логироваться.
- Конфигурационные файлы WordPress из `WORDPRESS__PATH`, включая `wp-config.php` и `.env`, входят в полный бэкап сайта.
- `.env` самого бота не должен попадать в резервную копию, если он находится вне `WORDPRESS__PATH`.
- Шифрование бэкапов в текущей версии не требуется.
- Восстановление должно требовать подтверждения.
- Команды дочерних процессов должны формироваться без shell-инъекций, через список аргументов.
- Команды WP-CLI должны формироваться без shell-инъекций, через список аргументов.
- Запуск WP-CLI от имени `www-data` должен выполняться без shell-инъекций, например через `runuser` со списком аргументов.
- Бот может запускаться от имени `root`, если это нужно для доступа к файлам сайта и бэкапам.
- Операции WP-CLI должны выполняться от имени `www-data` или пользователя из `WORDPRESS__CLI_RUN_AS_USER`, даже если основной процесс бота запущен от `root`.

## 18. Требования к реализации

Стек реализации:

- Python.
- `aiogram` для Telegram Bot API.
- `pydantic_settings` для загрузки и валидации конфигурации из переменных окружения и `.env`.
- SQLAlchemy ORM 2.0 с `asyncio` для работы с локальной БД.
- `APScheduler` для управления расписанием автоматических бэкапов.
- SQLite для локальной БД.
- `matplotlib` для построения графика дискового пространства.
- `numpy` и `pandas` как зависимости для обработки данных и построения графиков.
- `subprocess` для запуска локальных команд.

Интерфейс Telegram должен быть на русском языке. Поддержка i18n может быть добавлена позже, но не является обязательной для текущей версии.

Код должен быть разделен на модули:

- конфигурация;
- Telegram-интерфейс;
- авторизация и проверка ролей;
- сервис бэкапов;
- сервис восстановления;
- сервис планировщика;
- сервис проверки диска;
- сервис построения графиков;
- сервис очистки кэша WordPress;
- локальная БД;
- логирование;
- обработка ошибок.

## 19. Критерии приемки

Проект считается готовым, если:

- бот запускается с `.env`;
- неавторизованный пользователь не может выполнять команды;
- пользователи и роли берутся только из `TELEGRAM__SUPERADMIN_USERS`, `TELEGRAM__ADMIN_USERS`, `TELEGRAM__VIEWER_USERS`;
- бот не поддерживает управление пользователями через Telegram-команды;
- пользователь с недостаточной ролью не может выполнить запрещенную для него команду;
- пользователь с ролью `admin` или `superadmin` может создать полный бэкап;
- пользователь не может создать отдельно бэкап только файлов или только базы данных;
- полный бэкап создается как один `.tar.zst` архив;
- внутри `.tar.zst` архива есть `wordpress/` с файлами сайта и `database/db.sql` с дампом базы данных;
- после успешного бэкапа бот отправляет в Telegram полный локальный путь к бэкапу, но не отправляет сам файл бэкапа;
- планировщик автоматических бэкапов работает через `APScheduler`, настраивается только через cron-подобный конфиг и не управляется через Telegram;
- если плановый бэкап попал на уже выполняющийся backup/restore, он откладывается, а бот отправляет предупреждение администраторам;
- бот показывает список бэкапов;
- бот показывает свободное место на диске;
- бот отправляет круговую диаграмму диска через Telegram с долями: сервер, WordPress, бэкапы, свободное место;
- бот очищает кэш WordPress через WP-CLI командами `wp cache flush` и `wp transient delete --all`;
- WP-CLI команды очистки кэша выполняются от имени `www-data` или пользователя из `WORDPRESS__CLI_RUN_AS_USER`;
- при создании бэкапа или восстановлении бот переходит в статус выполняемой операции;
- бот не запускает две тяжелые операции одновременно и отказывает в новом backup/restore до завершения текущей операции;
- бот пишет историю операций в локальную БД;
- бот логирует ошибки системных команд;
- восстановление требует подтверждения;
- перед восстановлением создается аварийный бэкап;
- восстановление файлов выполняется через временную директорию;
- восстановление внешнего бэкапа работает через указание локального пути на сервере;
- есть инструкция по установке и запуску через `systemd`.

## 20. Допущения текущей версии

Пока ответы не уточнены, принимаются следующие допущения:

- бот обслуживает один WordPress-сайт;
- бот запускается на Ubuntu Server;
- WordPress установлен напрямую на сервере без Docker, виртуальной машины или контейнера;
- MySQL или MariaDB работают напрямую на сервере без Docker;
- пути `WORDPRESS__PATH` и `BACKUP__PATH_DIR` задаются через переменные окружения;
- резервные копии хранятся только локально;
- готовые бэкапы не отправляются файлами в Telegram;
- управление выполняется только через Telegram;
- интерфейс Telegram на русском языке;
- поддержка i18n может быть добавлена позже;
- локальная БД бота - SQLite;
- полный бэкап создается как один `.tar.zst` архив;
- внутри полного архива файлы сайта лежат как обычная папка `wordpress/`, а дамп базы данных как `database/db.sql`;
- исключения из архива не поддерживаются, в бэкап попадает весь `WORDPRESS__PATH`;
- конфигурационные файлы сайта внутри `WORDPRESS__PATH`, включая `wp-config.php` и `.env`, входят в бэкап;
- шифрование бэкапов не требуется;
- перед восстановлением создается аварийный бэкап;
- восстановление файлов выполняется через временную директорию;
- внешний бэкап для восстановления задается локальным путем на сервере;
- веб-сервер и PHP-FPM перед восстановлением не останавливаются;
- опасные операции требуют подтверждения;
- функция автоматической очистки старых бэкапов и retention-правила не реализуются;
- отдельная команда проверки целостности бэкапа не требуется;
- подробный журнал операций хранится только локально;
- очистка кэша WordPress выполняется через `wp cache flush` и `wp transient delete --all` от имени `www-data`;
- график диска строится как PNG через `matplotlib`, отправляется в Telegram и показывает круговую диаграмму;
- автоматическое расписание реализуется через `APScheduler` и управляется только через cron-подобную конфигурацию;
- плановое расписание по умолчанию: каждую пятницу в 03:00, `SCHEDULE__BACKUP_CRON="0 3 * * 5"`;
- если плановый бэкап наступает во время другой тяжелой операции, он откладывается, а бот отправляет предупреждение администраторам.
