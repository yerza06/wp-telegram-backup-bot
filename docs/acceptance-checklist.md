# Приемочный чеклист Telegram-WP-Backup

Источник: `docs/specs/technical-spec.md`.

## Запуск и конфигурация

- [ ] Бот запускается с `.env`.
- [ ] Python соответствует `>=3.13`.
- [ ] Конфигурация загружается через `pydantic-settings` и `__`-секции.
- [ ] Секреты не попадают в Telegram-сообщения и безопасные ошибки.
- [ ] Есть инструкция установки и запуска через `systemd`.

## Доступ

- [ ] Неавторизованный пользователь не может выполнять команды.
- [ ] Роли берутся только из `TELEGRAM__SUPERADMIN_USERS`, `TELEGRAM__ADMIN_USERS`, `TELEGRAM__VIEWER_USERS`.
- [ ] Нет команд управления пользователями.
- [ ] `viewer` может смотреть `/status`, `/disk`, `/disk_chart`, `/backups`.
- [ ] `admin` может запускать `/backup` и `/clear_cache`.
- [ ] Только `superadmin` может запускать restore.

## Backup

- [ ] `/backup` создает только полный бэкап.
- [ ] Нет команд file-only или db-only backup.
- [ ] Итоговый архив имеет формат `.tar.zst`.
- [ ] В архиве есть `wordpress/`, `database/db.sql`, `metadata.json`.
- [ ] SQL-дамп не создается как отдельный `.sql.gz`.
- [ ] Готовый архив не отправляется в Telegram файлом.
- [ ] Telegram показывает полный локальный путь к архиву.
- [ ] История операции записывается в SQLite.

## Restore

- [ ] Восстановление требует подтверждения.
- [ ] Перед restore создается аварийный бэкап.
- [ ] Restore из локального бэкапа использует запись в БД.
- [ ] Restore из внешнего архива принимает локальный путь сервера.
- [ ] Невалидный архив не меняет текущий сайт.
- [ ] Файлы восстанавливаются через временную директорию.
- [ ] БД восстанавливается через `mysql`.

## Диск, кэш, расписание

- [ ] `/disk` показывает total/used/free/use%/path.
- [ ] `/disk_chart` отправляет PNG с долями: сервер, WordPress, бэкапы, свободно.
- [ ] `/clear_cache` запускает `wp cache flush` и `wp transient delete --all`.
- [ ] WP-CLI запускается через `runuser` от `www-data` или настроенного пользователя.
- [ ] Планировщик APScheduler включается только через конфигурацию.
- [ ] Нет Telegram-команд управления расписанием.
- [ ] При активном backup/restore плановый бэкап откладывается.

## Безопасность и ошибки

- [ ] Команды системных процессов формируются списком аргументов.
- [ ] Пароль БД передается не через argv.
- [ ] Одновременно выполняется только один backup/restore.
- [ ] Ошибки системных команд логируются локально.
- [ ] Telegram получает короткие безопасные сообщения.

## Проверки

- [ ] `python -m compileall bot tests`
- [ ] `pytest -q`
- [ ] Локальная проверка ролей и handlers.
- [ ] Staging VPS проверка реального backup/restore согласно `docs/testing-strategy.md`.
