# Вопросы к техническому заданию

Документ содержит вопросы, которые нужно уточнить перед разработкой Telegram-бота для резервного копирования WordPress.

Цель документа - зафиксировать неоднозначности, технические риски и решения, которые должны быть явно приняты до реализации.

## 1. Консистентность резервной копии

1. Допускается ли создание резервной копии без перевода сайта в режим обслуживания?
2. Должен ли бот включать WordPress maintenance mode на время создания бэкапа?
3. Нужно ли блокировать обновления WordPress, плагинов и тем на время бэкапа?
4. Достаточно ли для БД использовать `mysqldump --single-transaction`?
5. Нужно ли поддерживать корректный бэкап MyISAM-таблиц, где `--single-transaction` не гарантирует консистентность?
6. Допустима ли ситуация, когда файлы WordPress и дамп БД сняты в немного разные моменты времени?
7. Нужно ли перед бэкапом проверять, что сейчас не выполняется обновление WordPress или плагинов?

## 2. Создание резервной копии

1. Как очищать временные директории внутри `BACKUP__TMP_PATH` после сбоев?
2. Нужно ли проверять свободное место с учетом примерного размера `WORDPRESS__PATH` и будущего SQL-дампа?
3. Что делать, если свободное место закончилось уже во время создания бэкапа?
4. Должен ли бот удалять staging-директорию после неуспешного бэкапа?
5. Нужно ли сохранять права файлов, владельцев, symlink-ы, hidden files, ACL и extended attributes?
6. Нужно ли использовать `tar --numeric-owner`?
7. Нужно ли проверять созданный `.tar.zst` архив после завершения бэкапа?
8. Должен ли `metadata.json` содержать версии WordPress, PHP, MySQL/MariaDB, список плагинов и тему?
9. Нужно ли записывать checksum итогового архива?
